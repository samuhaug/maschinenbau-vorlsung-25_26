Software Engineering

KISS, DRY und SRP – drei Prinzipien für guten Code und gute Systeme

01

Lernziele

📌 Nach dieser Vorlesung kannst du …

02

Was ist Software Engineering?

Software Engineering ist die systematische Anwendung von Ingenieursprinzipien auf die Entwicklung, Wartung und den Betrieb von Software. Es geht nicht nur darum, dass ein Programm funktioniert – es geht darum, dass es zuverlässig, wartbar und verständlich funktioniert, auch wenn es über Jahre hinweg von verschiedenen Menschen weiterentwickelt wird.

Die Parallele zum Maschinenbau ist direkt: Ein Maschinenbauingenieur baut nicht einfach irgendetwas, das funktioniert. Er achtet auf Materialwahl, Toleranzen, Montierbarkeit und Wartungsfreundlichkeit. Ein Software Engineer denkt genau so über seinen Code nach.

Was guten Code ausmacht

Guter Code ist korrekt, aber das allein reicht nicht. Er muss außerdem lesbar, testbar, änderbar und verständlich sein – auch für jemanden, der ihn sechs Monate später zum ersten Mal liest.

Was schlechten Code teuer macht

Schlechter Code funktioniert anfangs vielleicht – aber jede Änderung kostet mehr Zeit als die vorherige. Ein Fehler an einer Stelle bricht plötzlich etwas ganz anderes. Das Teamwork leidet, weil niemand den Code mehr durchblickt.

03

Technical Debt – Technische Schulden

Technical Debt (technische Schulden) bezeichnet den Mehraufwand, der durch bewusst oder unbewusst getroffene schlechte Designentscheidungen in der Vergangenheit entsteht. Wie bei einem finanziellen Kredit akkumulieren sich „Zinsen": Je länger schlechter Code existiert, desto aufwendiger wird es, ihn zu warten oder zu erweitern.

Anschauliches Beispiel

Ein Team baut in einer Fabrik schnell eine provisorische Steuerungslogik, um eine Deadline einzuhalten – „das klären wir später". Sechs Monate später hat die Maschine 12 weitere Erweiterungen, alle aufgebaut auf dem fragilen Provisorium. Jede neue Funktion dauert jetzt dreimal so lange und birgt das Risiko, bestehende Funktionen zu beschädigen. Das ist Technical Debt in der Praxis.

Bewusste Schulden

Manchmal nimmt man Technical Debt bewusst in Kauf – zum Beispiel für einen schnellen Prototyp. Entscheidend ist, dass man es weiß und einplant, die Schulden später zu begleichen.

Unbewusste Schulden

Häufiger entstehen Schulden unbewusst durch mangelndes Wissen oder schlechte Gewohnheiten. Genau hier helfen Prinzipien wie KISS, DRY und SRP – sie verhindern, dass Schulden überhaupt erst entstehen.

04

KISS – Keep It Simple, Stupid

Das KISS-Prinzip besagt: Wähle immer die einfachste Lösung, die die Anforderung erfüllt. Komplexität ist kein Zeichen von Qualität oder Intelligenz – im Gegenteil. Eine unnötig komplizierte Lösung ist schwerer zu verstehen, schwerer zu warten und fehleranfälliger als eine einfache.

Die Abkürzung wurde ursprünglich in den 1960er Jahren von der US Navy geprägt. Das Prinzip gilt für alle Bereiche des Engineerings, nicht nur für Software.

Die Ikea-Anleitung

Eine gute Montageanleitung zeigt jeden Schritt mit klaren Symbolen, ohne unnötige Erklärungen. Jeder kann sie befolgen – unabhängig von Vorkenntnissen. Das ist KISS: minimale Komplexität, maximale Verständlichkeit.

Das Roman-Problem

Stell dir vor, die Ikea-Anleitung wäre stattdessen ein 300-Seiten-Roman, der die Geschichte jedes Schraubendrehers beschreibt. Die Information ist da, aber komplett überwältigt von unnötiger Komplexität. Das passiert in Code, wenn Entwickler zu viel auf einmal lösen wollen.

KISS in der Praxis erkennen

Frage dich nach jeder Lösung: „Würde jemand, der das zum ersten Mal sieht, in 5 Minuten verstehen, was hier passiert?" Wenn die Antwort nein ist, gibt es eine einfachere Möglichkeit. Komplexität sollte aus dem Problem kommen, nicht aus der Lösung.

05

DRY – Don't Repeat Yourself

Das DRY-Prinzip besagt: Jede Information, jede Logik, jede Regel soll in einem System genau einmal vorhanden sein. Wer dieselbe Logik an zehn Stellen kopiert, muss sie bei einer Änderung auch an zehn Stellen aktualisieren – und vergisst garantiert mindestens eine. Das erzeugt Fehler und Inkonsistenzen.

Analogie: Die Adresse im Adressbuch

Angenommen, du schreibst einem Freund 10 verschiedene Briefe und trägst deine Rücksendeadresse in jeden Brief einzeln von Hand ein. Du ziehst um – und musst nun alle 10 Briefe suchen und korrigieren. Hättest du stattdessen ein Adressbuch geführt, hätte eine einzige Änderung genügt. DRY sagt: Leg das Adressbuch an.

Verletzung von DRY (Vorher)

Dieselbe Berechnung erscheint dreimal an verschiedenen Stellen. Wenn sich die Formel ändert, müssen alle drei Stellen gefunden und angepasst werden.

// Stelle 1: Berechne Preis mit Rabatt
preis = grundpreis * 0.9

// Stelle 2: Berechne anderen Preis
preis2 = andererPreis * 0.9

// Stelle 3: Nochmal dieselbe Logik
finalpreis = listenpreis * 0.9

DRY-konform (Nachher)

Die Rabatt-Logik ist an genau einer Stelle definiert. Eine Änderung des Rabattes erfordert nur eine einzige Anpassung – und alle Berechnungen sind automatisch konsistent.

// Die Logik lebt genau einmal
FUNKTION berechneRabattpreis(preis)
  ERGEBNIS = preis * 0.9
ENDE FUNKTION

preis1 = berechneRabattpreis(grundpreis)
preis2 = berechneRabattpreis(andererPreis)
preis3 = berechneRabattpreis(listenpreis)
Wann ist Wiederholung ein Warnsignal?

Sobald du den Verdacht hast, etwas kopiert zu haben, ist das ein Warnsignal. Die Frage lautet: „Wenn sich diese Logik ändert, wie viele Stellen müsste ich dann anpassen?" Wenn die Antwort mehr als eins ist, verletzt du DRY.

06

SRP – Single Responsibility Principle

Das Single Responsibility Principle (SRP, Prinzip der einzigen Verantwortlichkeit) besagt: Eine Funktion, ein Modul, eine Klasse soll genau eine klar definierte Aufgabe erfüllen – und diese gut. Wer eine Funktion schreibt, die gleichzeitig berechnet, speichert, formatiert und ausgibt, hat vier Aufgaben in einen Behälter gestopft. Das macht es fast unmöglich, eine davon zu ändern, ohne die anderen zu berühren.

Das Multitool-Problem

Ein Schweizer Taschenmesser kann viele Dinge – aber keines davon besonders gut. Wenn du wirklich gut schneiden musst, nimmst du ein richtiges Messer. Wenn du wirklich gut schrauben musst, nimmst du einen richtigen Schraubenzieher. Spezialisierung schlägt Multitasking in der Tiefe.

Spezialwerkzeug im Software Engineering

Eine Funktion mit einer Aufgabe ist wie ein gutes Spezialwerkzeug: leicht zu verstehen, leicht zu testen und leicht auszutauschen, ohne das gesamte System zu berühren. Mehrere kleine, klare Funktionen sind wartbarer als eine große, unüberschaubare.

SRP verletzt (Vorher)

Eine Funktion übernimmt alles: Einlesen, Berechnen und Ausgeben. Ändert sich das Ausgabeformat, muss man die gesamte Logik anfassen.

FUNKTION verarbeiteBestellung()
  // Daten einlesen
  EINGABE: menge, preis

  // Berechnung
  gesamt = menge * preis

  // Ausgabe formatieren und speichern
  AUSGABE: "Gesamt: " + gesamt
  Speichere in Datenbank
ENDE FUNKTION

SRP eingehalten (Nachher)

Drei Funktionen, je eine Verantwortlichkeit. Jede kann unabhängig verändert, getestet und ausgetauscht werden.

FUNKTION berechneGesamt(menge, preis)
  ERGEBNIS = menge * preis
ENDE FUNKTION

FUNKTION formatiereAusgabe(wert)
  ERGEBNIS = "Gesamt: " + wert
ENDE FUNKTION

FUNKTION speichereBestellung(wert)
  Speichere in Datenbank
ENDE FUNKTION
07

Zusammenfassung: Drei Prinzipien, ein Ziel

Merksatz

KISS sagt: Wähle die einfachste Lösung. DRY sagt: Wiederhole keine Logik. SRP sagt: Eine Funktion, eine Aufgabe. Alle drei Prinzipien verfolgen dasselbe Ziel: Code zu schreiben, der auch in einem Jahr noch verständlich, änderbar und zuverlässig ist. Technical Debt entsteht, wenn diese Prinzipien ignoriert werden – und wächst, bis Weiterentwicklung zum Stillstand kommt.

PrinzipKernaussageWarnsignalAnalogie
KISS So einfach wie möglich „Das versteht doch kein normaler Mensch" Ikea-Anleitung statt 300-Seiten-Roman
DRY Keine doppelte Logik „Das hab ich gerade copy-paste gemacht" Adressbuch statt Adresse 10 Mal schreiben
SRP Eine Aufgabe pro Einheit „Diese Funktion macht eigentlich 5 Sachen" Spezialwerkzeug statt Schweizer Taschenmesser
