02

Hash-Funktion – der digitale Fingerabdruck

Eine Hash-Funktion ist eine mathematische Funktion, die eine beliebig lange Eingabe nimmt und daraus eine Ausgabe fester Länge erzeugt – den sogenannten Hash-Wert oder kurz Hash. Der Hash ist wie ein digitaler Fingerabdruck: Er repräsentiert die Eingabe eindeutig, verrät aber nichts über den Originalinhalt.

Der fundamentale Unterschied zur Verschlüsselung: Eine Verschlüsselung ist umkehrbar – wer den Schlüssel hat, kann entschlüsseln. Eine Hash-Funktion ist nicht umkehrbar. Aus dem Hash kann man den Originalinhalt nicht zurückrechnen. Das macht Hash-Funktionen für bestimmte Anwendungen ideal – und für andere ungeeignet.

Hash-Funktion vs. Verschlüsselung

Verschlüsselung: „Hallo" → (mit Schlüssel K) → „Xklmq" → (mit Schlüssel K) → „Hallo" (umkehrbar, braucht Schlüssel zum Lesen). Hash-Funktion: „Hallo" → 185f8db32... – diese Operation ist nicht umkehrbar. Es gibt keinen Schlüssel, der zurück zu „Hallo" führt.

03

Die drei Kern-Eigenschaften einer guten Hash-Funktion

Nicht jede Funktion, die eine Eingabe in eine Zahl umwandelt, ist eine kryptografisch sichere Hash-Funktion. Drei Eigenschaften sind entscheidend:

1 – Deterministisch

Dieselbe Eingabe erzeugt immer denselben Hash-Wert. Das klingt trivial, ist aber fundamental: Nur so kann man später überprüfen, ob eine Eingabe mit einem gespeicherten Hash übereinstimmt. SHA-256("Hallo") = 185f8db32... – immer, überall, ohne Ausnahme.

2 – Einweg-Eigenschaft

Aus dem Hash-Wert lässt sich die Eingabe nicht zurückrechnen. Die Berechnung geht nur in eine Richtung. Selbst mit unbegrenzter Rechenzeit gibt es keinen effizienten Algorithmus, der aus 185f8db32... das ursprüngliche „Hallo" rekonstruiert.

3 – Lawineneffekt (Avalanche Effect)

Eine minimale Änderung der Eingabe führt zu einem völlig anderen Hash-Wert. Aus „Hallo" wird ein komplett anderer Hash als aus „Hallo!" (mit Ausrufezeichen). Diese Eigenschaft verhindert, dass Angreifer aus ähnlichen Eingaben auf ähnliche Hashes schließen können.

04

Hash-Algorithmen im Vergleich

Im Laufe der Zeit wurden verschiedene Hash-Algorithmen entwickelt. Einige gelten heute als unsicher, weil Angreifer Kollisionen (zwei verschiedene Eingaben mit gleichem Hash) erzeugen konnten. Für neue Systeme gibt es klare Empfehlungen.

AlgorithmusHash-LängeStatusEinsatz
MD5 128 Bit (32 Zeichen hex) ❌ Unsicher Nicht mehr für Sicherheitszwecke; höchstens noch für Prüfsummen ohne Sicherheitsrelevanz
SHA-1 160 Bit (40 Zeichen hex) ❌ Unsicher 2017 praktisch gebrochen; nicht mehr verwenden
SHA-256 256 Bit (64 Zeichen hex) ✅ Empfohlen Standard für Passwort-Hashing, Blockchain (Bitcoin), TLS-Zertifikate, digitale Signaturen
SHA-3 variabel (224–512 Bit) ✅ Empfohlen Modernster Standard (2015); andere interne Struktur als SHA-2 als Backup-Option
Kollisionsangriffe – eine Randnotiz

Bei MD5 und SHA-1 konnten Forscher gezielt zwei verschiedene Dokumente mit gleichem Hash-Wert erzeugen – das nennt sich Kollision. Damit lassen sich z. B. gefälschte Zertifikate erstellen. SHA-256 gilt aktuell als kollisionsresistent.

05

Passwort-Hashing – warum nie im Klartext speichern?

Ein Server muss überprüfen können, ob du das richtige Passwort eingibst – aber er darf dein Passwort selbst nicht kennen. Die Lösung: Der Server speichert nicht das Passwort, sondern nur dessen Hash-Wert. Beim Login berechnet der Server den Hash des eingegebenen Passworts und vergleicht ihn mit dem gespeicherten Hash. Stimmen sie überein, ist das Passwort korrekt – ohne dass das Klartext-Passwort je gespeichert oder übertragen wird.

Praxisbeispiel: haveibeenpwned.com

Auf der Website haveibeenpwned.com kann man prüfen, ob die eigene E-Mail-Adresse in einem bekannten Datenbankleaks aufgetaucht ist. Die Website speichert Hash-Werte von hunderten Millionen gestohlener Passwörter. Wenn ein Angreifer eine Datenbank mit Passwort-Hashes stiehlt, kann er aus den Hashes nicht direkt die Passwörter lesen – er muss jeden Hash einzeln „knacken".

Rainbow Tables – der nächste Angriff

Wenn alle Nutzer mit gleichem Passwort denselben Hash haben, kann ein Angreifer sogenannte Rainbow Tables nutzen: vorberechnete Tabellen mit Millionen von Passwörtern und deren Hashes. Steht ein Hash in der Tabelle, ist das Passwort sofort bekannt. Lösung: Salt.

06

Salt – individuelles Zufallsgewürz pro Passwort

Ein Salt ist eine zufällig generierte Zeichenkette, die vor dem Hashing an das Passwort angehängt wird. Jeder Nutzer bekommt ein eigenes, zufälliges Salt. Das Salt wird nicht geheim gehalten – es wird zusammen mit dem Hash in der Datenbank gespeichert. Trotzdem macht es Rainbow-Table-Angriffe praktisch wirkungslos, weil Rainbow Tables für ein bestimmtes Salt vollständig neu berechnet werden müssten.

07

Digitale Signatur – Echtheit und Integrität beweisen

Eine digitale Signatur ist das digitale Äquivalent einer handschriftlichen Unterschrift – aber deutlich fälschungssicherer. Sie beweist gleichzeitig zwei Dinge: dass eine Nachricht oder Datei von einem bestimmten Absender stammt (Authentizität) und dass sie seit dem Signieren nicht verändert wurde (Integrität).

Das technische Prinzip kombiniert Hash-Funktion und asymmetrische Kryptografie: Hash-Wert der Nachricht + Signierung mit Private Key = Digitale Signatur. Der Empfänger berechnet selbst den Hash der empfangenen Datei und entschlüsselt die beigelegte Signatur mit dem öffentlichen Schlüssel des Absenders. Stimmen beide Hashes überein, ist die Datei authentisch und unverändert.

Praxisbeispiel: Firmware-Update-Verifikation

Wenn du ein Firmware-Update für deinen Router oder ein Softwareupdate für deinen Laptop herunterlädst, wird die Datei mit einer digitalen Signatur des Herstellers geliefert. Dein Gerät überprüft die Signatur automatisch: Wurde die Datei seit dem Signieren durch den Hersteller verändert? Stimmt der Hash? Nur wenn beide Checks positiv sind, wird das Update installiert. Das verhindert, dass ein Angreifer manipulierte Firmware einschleusen kann – selbst wenn er die Verbindung abfängt.

Digitale Signatur ≠ Verschlüsselung

Eine Signatur verschlüsselt den Inhalt nicht – jeder kann die signierte Datei lesen. Die Signatur beweist nur Echtheit und Unversehrtheit, nicht Vertraulichkeit. Für Vertraulichkeit braucht man zusätzlich Verschlüsselung.

08

Zusammenfassung: Kryptografie Teil 2

Merksatz

Eine Hash-Funktion ist eine Einwegfunktion – aus dem Hash kann man die Eingabe nicht zurückrechnen. Gute Hash-Funktionen sind deterministisch, haben die Einweg-Eigenschaft und zeigen den Lawineneffekt. SHA-256 ist der aktuelle Standard; MD5 und SHA-1 sind gebrochen. Passwörter werden als Hash gespeichert – nie im Klartext. Ein Salt macht Rainbow-Table-Angriffe wirkungslos. Digitale Signaturen beweisen Authentizität und Integrität durch Hash + Private Key.

KonzeptKernaussage
Hash-FunktionEinweg; beliebige Eingabe → fixer Hash; nicht umkehrbar
DeterministischGleiche Eingabe → immer gleicher Hash
Einweg-EigenschaftHash → Eingabe nicht zurückrechenbar
LawineneffektMinimale Änderung der Eingabe → völlig anderer Hash
SHA-256Aktueller Standard; MD5 und SHA-1 sind veraltet und unsicher
Passwort-HashingNur Hash wird gespeichert; Klartext bleibt unbekannt
SaltZufallszeichenkette pro Nutzer; macht Rainbow Tables wirkungslos
Digitale SignaturHash + Private Key = Beweis für Authentizität und Integrität
